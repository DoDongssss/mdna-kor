# 📁 Motorcycle Club System — Folder Structure

> **Stack:** React + TypeScript · Supabase · Tailwind CSS · shadcn/ui · Zustand · React Hook Form + Zod · Recharts

---

## Root

```
mc-club-system/

```

## `src/supabase/` — Backend Client & Types

```
src/supabase/
├── client.ts                   # createClient() singleton
├── migrations/                 # SQL migration files (versioned)
│   ├── 001_init_schema.sql
│   ├── 002_rls_policies.sql
│   └── 003_seed_data.sql
└── types/
    └── database.types.ts       # Auto-generated via supabase gen types
```

> Run `supabase gen types typescript --linked > src/supabase/types/database.types.ts`
> to regenerate types whenever the schema changes.

## `src/types/` — Domain TypeScript Interfaces

```
src/types/
├── members.ts                  # Member, MemberInsert, MemberUpdate
├── eyeballs.ts                 # Eyeball, EyeballInsert
├── attendance.ts               # Attendance, AttendanceStatus
├── contributions.ts            # Contribution, ContributionInsert
├── expenses.ts                 # Expense, ExpenseInsert
└── events.ts                   # ClubEvent, ClubEventInsert
```

> These are **hand-written** domain types (not auto-generated).
> They extend or map from `database.types.ts` for safety.

## `src/schemas/` — Zod Validation Schemas

```
src/schemas/
├── member.schema.ts            # createMemberSchema, updateMemberSchema
├── eyeball.schema.ts           # eyeballSchema
├── contribution.schema.ts      # contributionSchema
├── expense.schema.ts           # expenseSchema
└── event.schema.ts             # clubEventSchema
```

> Schemas are used with `React Hook Form` via `zodResolver()`.
> Keep schema files co-located with types — one domain, two concerns.

---

## `src/lib/` — Data Access Layer (Supabase Queries)

```
src/lib/
├── members.ts                  # getMembers, createMember, toggleActive, softDelete
├── eyeballs.ts                 # getEyeballs, getEyeballById, createEyeball
├── attendance.ts               # getAttendance, upsertAttendance
├── contributions.ts            # getContributions, addContribution
├── expenses.ts                 # getExpenses, addExpense, deleteExpense
└── events.ts                   # getEvents, createEvent, deleteEvent
```

> **Rule:** Components never call Supabase directly.
> All DB interactions go through `lib/` functions.
> This makes the codebase testable and backend-agnostic.

## `src/stores/` — Global State (Zustand)

```
src/stores/
├── auth.store.ts               # session, user, isAdmin, login, logout
├── members.store.ts            # members[], isLoading, fetchMembers
└── fund.store.ts               # totalBalance, contributions[], expenses[]
```

> Use Zustand for cross-cutting state (auth, fund totals).
> Use local React state or React Query for page-level data.

## `src/hooks/` — Custom Hooks

```
src/hooks/
├── useAuth.ts                  # Wraps auth.store, checks session on mount
├── useMembers.ts               # Fetches + caches member list
├── useEyeballs.ts              # Fetches eyeballs with pagination
├── useFund.ts                  # Calculates balance from contributions - expenses
├── useAttendance.ts            # Fetch + upsert attendance per eyeball
└── useEvents.ts                # Fetch public events list
```

> They are the bridge between data and UI.

## Key Conventions

- **No direct Supabase calls in components** — always go through `src/lib/`
- **One Zod schema per domain** — used for both form validation and type inference
- **Soft delete on members** — use `deleted_at` timestamp, never hard delete
- **`eyeball_id` is nullable on contributions** — supports out-of-meetup donations
- **`eyeball_id` is nullable on expenses** — not all expenses belong to a meetup
- **Role check via `auth.store.ts`** — `isAdmin` flag gates all admin routes
